# Аркадна гра

## Завдання

### Завдання 1: Виявлення порушень принципів SOLID

1) Порушено принципи DRY (дублювання коду)

- У класі `GameScene`, `Zombie` та `Imp` дублюється метод `shoot()`

2) Порушено принип SRP (одна відповідальність)

- Клас `GameScene` виконує декілька різних обов'язків:
    - Керує запуском рівня через метод `start()`
    - Управляє монстрами через метод `addMonster()`
    - Зберігає монстрів у списку `zombies`
  
3) Погана масштабованість
- Три списки (`demons`/`imps`/`zombies`)

4) Open/Closed принцип порушено
- Щоб додати новий тип монстра, треба міняти MonsterType, addMonster() (switch) і ще списки/цикл у start(). Код не “закритий для змін”.

5) Dependency Inversion порушено
- `GameScene` залежить від конкретних класів монстрів (`Zombie`, `Imp`, `Demon`), а мав би залежати від абстракції.

### Завдання 2: Рефакторинг коду

1) Додавати нові типи монстрів зараз дуже незручно, тому що треба міняти код у декількох місцях.
- Щоб додати нового монстра, треба:
  - Додати нове значення в `MonsterType`.
  - Створити новий клас монстра.
  - Додати новий кейс у свіч у `addMonster()`.
  - Додати новий цикл у `start()` для виклику `shoot()`.
  - Опційно! додати новий список для зберігання монстрів цього типу.
- Це порушується принцип Open/Closed, про що писав у першому завданні, бо для розширення функціоналу треба редагувати існуючий код.

2) У першу чергу створив би інтерфейс або абстрактний клас, щоб `GameScene` міг працювати з монстрами через абстракцію, а не конкретні класи. Це дозволить додавати нові типи монстрів без зміни коду `GameScene`.
